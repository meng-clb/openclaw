# 2026-03-12 钉钉安全与 MCP 权限调整日志

## 一、变更背景

本次调整源于以下两个直接需求：

1. 需要收紧 OpenClaw 对钉钉消息的权限入口，避免未经授权的私聊或群聊直接进入高权限工作区。
2. 需要保留钉钉 AI 表格 MCP 能力，允许群内用户继续操作钉钉侧的多维表数据，但不能再借此修改宿主机本地文件、配置或执行任意命令。

在调整前，钉钉侧存在以下风险：

- 钉钉私聊入口为开放状态，未限制发送者范围。
- 钉钉群聊默认可落到高权限工作区，存在被提示词诱导后改本地文件、执行命令、泄露敏感信息的风险。
- 历史群会话可能继续复用旧上下文，造成“规则已改但旧会话仍有高权限”的错觉。

## 二、本次完成的变更

### 1. 收紧钉钉私聊入口

变更文件：

- `/root/.openclaw/openclaw.json`

变更内容：

- 将 `channels.dingtalk.dmPolicy` 从开放模式改为 `allowlist`
- 将 `channels.dingtalk.allowFrom` 收紧为仅允许 `161054433038257862`

变更原因：

- 只允许明确授权的私聊用户进入高权限工作区，防止其他钉钉用户通过私聊直接触发高权限操作。

### 2. 将钉钉群聊与高权限私聊彻底分流

变更文件：

- `/root/.openclaw/openclaw.json`
- `/root/.openclaw/workspace-ddtalk-group/AGENTS.md`
- `/root/.openclaw/workspace-ddtalk-group/SOUL.md`
- `/root/.openclaw/workspace-ddtalk-group/USER.md`
- `/root/.openclaw/workspace-ddtalk-group/TOOLS.md`

变更内容：

- 新增受限群聊工作区 `workspace-ddtalk-group`
- 新增精确路由：
  - 钉钉私聊且发送者为 `161054433038257862` 时，进入高权限工作区 `workspace-ddtalk`
  - 其他钉钉消息默认进入受限工作区 `workspace-ddtalk-group`

变更原因：

- 将“本人私聊高权限”和“群聊低权限”从路由层分开，而不是仅依赖提示词约束。
- 即使群聊内出现恶意诱导，请求也不会直接进入高权限工作区。

### 3. 归档并清空旧钉钉会话

变更路径：

- 归档目录：`/root/.openclaw/agents/workspace-ddtalk/sessions-archive/manual-reset-20260312-204126/`
- 重建后的会话索引：
  - `/root/.openclaw/agents/workspace-ddtalk/sessions/sessions.json`
  - `/root/.openclaw/agents/workspace-ddtalk-group/sessions/sessions.json`

变更内容：

- 将旧钉钉会话归档
- 清空高权限与受限群聊工作区当前活动会话索引
- 保留归档，便于后续审计与回溯

变更原因：

- 防止旧群会话继续绑定到高权限上下文
- 确保后续新消息全部按照新路由与新权限体系进入新会话

### 4. 将群聊能力从“完全禁用”调整为“允许受限 MCP，不允许本地文件操作”

变更文件：

- `/root/.openclaw/openclaw.json`
- `/root/.openclaw/bin/dingtalk-mcp-ro`
- `/root/.openclaw/workspace-ddtalk-group/AGENTS.md`
- `/root/.openclaw/workspace-ddtalk-group/TOOLS.md`
- `/root/.openclaw/workspace-ddtalk-group/skills/dingtalk-ai-table-group/SKILL.md`

变更内容：

- 群聊工作区工具策略改为：
  - 允许：`session_status`、`exec`
  - 禁止：`read`、`write`、`edit`、`apply_patch`、`process`
- 但这里的 `exec` 不是通用 shell，而是受限执行：
  - `host = gateway`
  - `security = allowlist`
  - `ask = off`
  - `safeBins = ["dingtalk-mcp-ro"]`
- 新增包装器脚本 `dingtalk-mcp-ro`
  - 只允许 `schema`
  - 只允许 `call <tool_name> [--args <json>]`
  - 固定服务器名为 `dingtalk-ai-table`
  - 明确拒绝 `config`、`auth`、`daemon`、文件生成等本地状态改写型命令
- 新增群聊专用 skill：`dingtalk-ai-table-group`
  - 要求群聊只通过 `dingtalk-mcp-ro` 调用钉钉 AI 表格 MCP
  - 禁止在该工作区直接使用原始 `mcporter`

变更原因：

- 用户明确要求“群里可以调用 MCP，但不能改我的本地文件”
- 仅禁用 `read/write/edit` 不足以满足安全目标，因为通用 `exec` 仍可通过 shell 直接改文件
- 因此需要采用“受限 exec + allowlist + 安全包装器”的组合方式，只保留钉钉 AI 表格 MCP 所需能力，不开放宿主机任意命令执行能力

## 四、当前安全边界

调整后的真实边界如下：

- 本人钉钉私聊：可以进入高权限工作区
- 钉钉群聊：不能改宿主机本地文件，不能跑任意 shell，不能使用文件编辑工具
- 钉钉群聊：可以通过受限包装器调用钉钉 AI 表格 MCP
- 钉钉群聊：仍然可以改“钉钉远端表格数据”，因为这是 MCP 本身提供的业务能力，不属于宿主机文件权限
